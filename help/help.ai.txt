SHARD — ENCRYPTED KNOWLEDGE ROUTING SYSTEM

You are connected to a shard hub. The hub is the MCP process that manages a network of shard nodes. Each node is a self-contained encrypted binary that holds thoughts (encrypted knowledge units).

ARCHITECTURE:
  Hub: The MCP process (this). Stays alive, routes requests, reads any shard.
  Nodes: Individual shard binaries. Ephemeral — spawn on demand, exit after idle timeout.
  Index: ~/.shards/index/ lists all registered shards.

HOW TO USE:
  1. shard_list — see all shards in the network
  2. shard_ask — ask a specific shard a question (use shard param to target)
  3. fleet_ask — ask ALL shards, get merged answers
  4. shard_write — store knowledge in a shard
  5. shard_query — keyword search within a shard
  6. shard_ingest — send raw data, LLM decomposes using descriptors
  7. create_shard — create a new shard node

ROUTING:
  Every shard has gates (semantic descriptions) and descriptors (format schemas).
  The hub checks gates via vector similarity before reading encrypted content.
  Use fleet_ask when you don't know which shard has the answer.
  Use shard_ask with a shard param when you know the target.

DATA MODEL:
  Thoughts are encrypted with ChaCha20-Poly1305, HKDF per-thought key derivation.
  Each thought has: id, description, content, agent, timestamps, trust token.
  Losing the binary means losing the data. No backup by design.

PROTOCOL:
  IPC uses length-prefixed JSON over Unix domain sockets.
  MCP uses JSON-RPC 2.0 over stdio.
