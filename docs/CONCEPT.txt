--------------------------------------------------------------------------------
ONE-SENTENCE DEFINITION
--------------------------------------------------------------------------------

A shard is an encrypted thought store loaded in-process by a daemon that
acts as the nervous system for a self-improving second brain.


--------------------------------------------------------------------------------
WHAT THIS IS NOT
--------------------------------------------------------------------------------

- Not a database with rows and queries
- Not a flat key-value store
- Not a theoretical protocol spec — this is built and running
- Not a distributed mesh — it runs on one machine, one process


--------------------------------------------------------------------------------
WHAT THIS IS
--------------------------------------------------------------------------------

An encrypted knowledge system designed for AI agents and humans to share.

Knowledge enters as raw thoughts — encrypted, timestamped, attributed to
whoever wrote them. Over time, an AI compacts those thoughts into ordered,
refined knowledge. The output is Markdown: YAML frontmatter for metadata,
wikilinks for connections, clean prose for content. Save a response to a
file and open it in Obsidian.

The architecture is one EXE, one daemon process. Each topic gets its own
shard file with its own encryption key. The daemon loads shard blobs
in-process on demand, evicts idle ones, and persists each to a single
binary file. The daemon is the discovery layer and the routing layer.

The wire protocol is Markdown with YAML frontmatter — the same format as
the final output. No translation layer between what the AI sees and what
the human browses.


--------------------------------------------------------------------------------
CORE MENTAL MODEL
--------------------------------------------------------------------------------

shard := catalog                 (plaintext identity card)
       + gates                   (routing signals: what I want / don't want)
       + processed thoughts      (AI-ordered, compacted knowledge)
       + unprocessed thoughts    (append-only buffer of new inputs)
       + manifest                (freeform metadata)

daemon := registry               (every shard: name, path, catalog, gates)
        + slots                  (in-process loaded shard blobs)
        + vector index           (shard embeddings for semantic routing)
        + event hub              (notify/events ops, auto-emit on write/compact)

Each thought is:
  - Encrypted with ChaCha20-Poly1305 under a per-thought key derived
    from the shard's master key via HKDF-SHA256
  - Attributable: tagged with who wrote it and when
  - Searchable: description is indexed for keyword search without decryption
  - Sealed: encrypted SHA256 of the description for integrity verification


--------------------------------------------------------------------------------
KEY ENTITIES
--------------------------------------------------------------------------------

THOUGHT
  The atomic unit. An encrypted note with a description and content body.
  Has:
    - id             — 16 random bytes, hex-encoded on the wire
    - description    — searchable summary (encrypted in storage)
    - content        — full body (encrypted in storage)
    - agent          — who wrote this (plaintext, max 64 chars)
    - created_at     — RFC3339 timestamp (plaintext)
    - updated_at     — RFC3339 timestamp (plaintext)
    - revises        — ID of parent thought this revises (zero = original)
    - seal           — encrypted SHA256 of description (verification)
    - trust token    — SHA256(key || SHA256(plaintext)) (integrity)
    - ttl              — staleness TTL in seconds (0 = immortal, plaintext)
    - read_count       — times read by agents (plaintext, for relevance scoring)
    - cite_count       — times cited by other thoughts (plaintext, for relevance scoring)

SHARD
  A persistent encrypted thought store. One file, one key.
  Loaded in-process by the daemon as a Shard_Slot on demand.
  Has:
    - name           — human-readable, determines IPC endpoint
    - master key     — 32 bytes, supplied as 64-char hex
    - catalog        — identity card (name, purpose, tags, related shards)
    - gates          — routing signals (description, positive, negative)
    - processed      — AI-ordered thoughts (compacted knowledge)
    - unprocessed    — append-only buffer (raw inputs)
    - manifest       — freeform metadata

DAEMON
  The nervous system. A single process at a fixed IPC endpoint that loads
  and manages all shard blobs in-process via Shard_Slots.
  Has:
    - registry       — every shard ever seen (name, path, thought count,
                       catalog, gates — cached for AI routing)
    - slots          — in-process loaded shard blobs (lazy load, idle eviction)
    - vector index   — shard embeddings for semantic routing
    - audit trail    — content alert review history (informational only)
    - event hub      — shards notify each other of changes. Events route
                       through catalog.related links. Origin chains prevent
                       circular propagation. Events queue per target shard
                       and clear on retrieval.
    - consumption log — per-agent, per-shard access tracking. Ring buffer
                        of 1000 records. Persisted to .shards/.consumption.
                        Powers gap detection (needs_attention flag).

CATALOG
  A shard's plaintext identity card. Readable without the encryption key.
  Fields: name, purpose, tags, related (links to other shards), created.
  Used by the daemon for discovery and by agents for routing decisions.

GATES
  Plaintext routing signals on each shard. Four lists:
    - description    — what this shard is about
    - positive       — terms this shard is drawn to (wants to receive)
    - negative       — terms this shard rejects (does not want)
    - related        — names of related shards (for cross-shard traversal)
  Agents read gates to decide which shards are relevant to a task.
  The daemon caches gates in the registry for fast AI-driven routing.


--------------------------------------------------------------------------------
TWO-BLOCK DESIGN
--------------------------------------------------------------------------------

Every shard has two thought blocks:

  UNPROCESSED — append-only buffer
    New thoughts land here. No ordering. Just a growing list of raw inputs
    from agents. This is the inbox.

  PROCESSED — AI-ordered knowledge
    When an AI reviews a shard, it reads everything, decides the best order,
    and issues a compact operation that moves thoughts from unprocessed to
    processed in the specified order. This is the curated knowledge base.

The compact operation is the intelligence step. The AI decides what matters,
what order it should be in, and what to keep. Over time, the processed block
becomes increasingly refined — a distilled, structured understanding of the
topic. The unprocessed block is just the queue of new inputs waiting for
the next review cycle.


--------------------------------------------------------------------------------
WIRE PROTOCOL
--------------------------------------------------------------------------------

All communication uses Markdown with YAML frontmatter over local IPC.
Length-prefixed: [u32 LE: payload_length][payload: UTF-8 markdown bytes].

Requests:
  ---
  op: write
  description: database schema patterns
  agent: code-reviewer
  ---
  Normalized table structure with composite indexes...

Responses:
  ---
  status: ok
  description: database schema patterns
  agent: code-reviewer
  created_at: 2026-03-15T19:32:28Z
  ---
  Normalized table structure with composite indexes...

The body after the closing --- is always the content field. Everything
else lives in the frontmatter as key: value pairs. Lists use inline
YAML syntax: tags: [design, patterns, sql].

This format is Obsidian-native. A read response is a valid .md file.
The dump operation returns an entire shard as one Obsidian document with
YAML frontmatter, [[wikilinks]] to related shards, and each thought
rendered as a ### heading with its content.

IPC endpoints:
  Windows:    \\.\pipe\shard-<name>
  Linux/macOS: /tmp/shard-<name>.sock
  Daemon:     fixed name "shard-daemon" (no key needed)


--------------------------------------------------------------------------------
ENCRYPTION
--------------------------------------------------------------------------------

Simple, symmetric, per-shard.

  - Master key: 32 bytes, supplied as 64-char hex or SHARD_KEY env var
  - Per-thought key: HKDF-SHA256(master, thought_id)
  - Cipher: ChaCha20-Poly1305 (IETF, 96-bit nonce, 128-bit tag)
  - Body: description + "\n---\n" + content (encrypted as one blob)
  - Seal: encrypted SHA256(description) for verification without full decrypt
  - Trust: SHA256(key || SHA256(plaintext)) for integrity binding

No asymmetric crypto. No key hierarchy. No per-entry access control.
One key per shard. You have the key or you don't.

Agent identity and timestamps are stored as plaintext metadata alongside
the encrypted body — queryable without decryption, but only meaningful
if you can also decrypt the content.


--------------------------------------------------------------------------------
PROCESS LIFECYCLE
--------------------------------------------------------------------------------

  0. First time: `shard init` creates .shards/, keychain, config
  1. AI connects via MCP (`shard mcp`) — daemon is auto-started if not running
  2. Sends registry/discover — gets list of all known shards with catalogs
  3. Reads gates and catalogs to decide which shards are relevant
  4. Sends ops with name: <shard> — daemon routes to the in-process slot
     - If slot not loaded: daemon loads the .shard file on demand
     - Key provided per-request or resolved from .shards/keychain
  5. Operations: write, read, search, query, compact, dump, etc.
  6. Disconnect when done
  7. Idle slots are evicted (flushed to disk, memory released)
  8. Daemon persists registry to its own .shard file

The daemon never dies (no timeout). The MCP server auto-starts it on
first connection if it isn't already running. It persists the registry
as JSON in its own blob's manifest field. On restart, it loads the
registry and scans .shards/ for new or changed shard files.

Concurrency: each IPC connection runs in its own thread. A mutex on
the Node serializes dispatch calls (all shared state access). I/O
(recv/send) happens outside the lock, so multiple MCP sessions or
connect clients can be active simultaneously.

Standalone shard mode still exists for remote/federated use:
  shard --name X --key K --data X.shard
This runs a single shard as its own process with its own IPC endpoint.


--------------------------------------------------------------------------------
FILE FORMAT (.shard)
--------------------------------------------------------------------------------

A standalone binary blob. One file per shard.

  [PROCESSED BLOCK]      count-prefixed binary thoughts (with revises + TTL + counters)
  [UNPROCESSED BLOCK]    count-prefixed binary thoughts (with revises + TTL + counters)
  [CATALOG BLOCK]        length-prefixed plaintext JSON
  [MANIFEST BLOCK]       length-prefixed plaintext
  [GATES BLOCK]          plaintext routing signals (4 lists: desc, pos, neg, related)
  [gates_size: u32 LE]   4 bytes
  [blob_hash: u8x32]     SHA-256 of all preceding bytes
  [MAGIC: u64 LE]        0x5348524430303036 ("SHRD0006")

Current format: SHRD0006 (adds read_count + cite_count fields to each thought).
Migrates SHRD0005 (TTL but no counters) and SHRD0004 (no TTL, no counters) on load.

Writes are atomic: write to temp file, verify hash, rename over original.
Footer is 44 bytes. Magic number identifies format version.


--------------------------------------------------------------------------------
WHAT IS VISIBLE WITHOUT THE KEY
--------------------------------------------------------------------------------

Without the encryption key, you can see:

  - Catalog: shard name, purpose, tags, related shards, creation date
  - Gates: description, positive signals, negative signals
  - Manifest: freeform metadata
  - Thought count (processed and unprocessed)
  - Agent names and timestamps on each thought
  - File size and structure

You CANNOT see:

  - Thought descriptions (encrypted)
  - Thought content (encrypted)
  - What any thought actually says

The plaintext metadata is deliberately exposed for routing — agents need
to discover and evaluate shards without holding every key.


--------------------------------------------------------------------------------
OBSIDIAN INTEGRATION
--------------------------------------------------------------------------------

The system is designed to export to Obsidian as a first-class output.

The dump operation decrypts all thoughts in a shard and returns a single
Markdown document:

  ---
  title: Architecture
  purpose: System design and patterns
  tags: [design, patterns]
  related: [rate-limiting, authentication]
  created: 2026-03-15T10:00:00Z
  exported: 2026-03-15T19:32:28Z
  thoughts: 15
  ---

  # Architecture

  System design and patterns

  ## Related
  [[rate-limiting]] | [[authentication]]

  ## Knowledge

  ### Database Schema Design
  Thoughts on table structure, indexes...

  ### Event-Driven Architecture
  Patterns for reactive systems...

  ## Unprocessed

  ### API Rate Limiting Notes
  Token bucket vs sliding window...

Save this to a .md file. Open in Obsidian. The graph view shows
connections via [[wikilinks]]. Tags are discoverable. Frontmatter
is indexed. The second brain is human-browsable.


--------------------------------------------------------------------------------
WHERE THIS IS HEADED
--------------------------------------------------------------------------------

MILESTONE 1: ATTRIBUTABLE KNOWLEDGE (complete)
  Every thought is traceable — who wrote it, when, and what it revised.
  Shards link to related shards. Multiple agents can write safely.
  All 5 specs done: agent-identity, slot-linking, conflict-resolution,
  atomic-memory-access (access op, transactions, write queuing),
  sensitive-content-alert (AI-based scanner, informational alerts, audit trail).
  Content scanning never blocks writes — the AI evaluates post-write and
  raises informational alerts for user review.

MILESTONE 2: INTELLIGENT CONSUMPTION (in progress)
  Agents follow a standardized read-evaluate-contribute cycle. Knowledge
  has freshness signals. Shards are ranked by relevance.
  Status: daemon-event-hub done (notify, events, auto-emit on write/compact).
  New: digest op (compressed knowledge base overview) and budget-limited
  queries (cap response size, truncate to fit) reduce agent token costs.
  New: staleness-ttl done (TTL field, staleness scoring, stale op, freshness-weighted search).
  New: relevance-scoring done (read/cite counters, composite scoring formula,
  feedback op, configurable weights, SHRD0006 format with V5/V4 migration).
  New: layered-shard-traversal done (3-layer progressive search: gates → thoughts → related shards,
  daemon-level multi-shard search, composite scoring, budget truncation).
  Remaining: fleet-dispatch.

MILESTONE 3: UNIFIED KNOWLEDGE BASE
  Cross-shard queries: one question searches everything. Bulk export to
  Obsidian vault. The system feels like one mind, not a collection of files.

MILESTONE 4: SELF-COMPACTING INTELLIGENCE
  The system refines its own knowledge automatically. An AI periodically
  reviews each shard, merges revision chains, prunes contradictions, and
  distills topics to their essence. Each cycle makes the system more precise.


--------------------------------------------------------------------------------
IMPLEMENTATION
--------------------------------------------------------------------------------

Language: Odin (https://odin-lang.org/)
Platform: Windows (named pipes), Linux/macOS (Unix sockets)
Binary:   Single EXE — runs as daemon, shard, MCP server, or CLI tool

Source files:
  src/main.odin          Entry point, CLI parsing, subcommands
  src/types.odin         Core types: Thought (with counters), Blob, Node, Catalog, Request/Response
  src/crypto.odin        HKDF, ChaCha20-Poly1305, thought create/decrypt/verify, binary serialization (SHRD0006)
  src/blob.odin          .shard file format (SHRD0006): load, flush, atomic write, V4/V5 migration
  src/daemon.odin        Registry, in-process slots, routing, layered traverse (L0/L1/L2), transactions, digest, staleness, feedback
  src/protocol.odin      Op dispatch: write, read, search, query, compact, dump, gates, stale, feedback, composite scoring
  src/markdown.odin      YAML frontmatter parser/serializer for wire format
  src/mcp.odin           MCP server: JSON-RPC 2.0 over stdio, 10 tools, daemon auto-start
  src/embed.odin         Vector embeddings via OpenAI-compatible API, cosine similarity
  src/config.odin        .shards/config reader with defaults (includes relevance weights)
  src/keychain.odin      .shards/keychain reader (shard name to key mapping)
  src/node.odin          Process lifecycle, event loop, idle timeout/eviction
  src/ipc.odin           Platform-neutral message framing (u32 LE length prefix)
  src/ipc_windows.odin   Windows named pipes (overlapped I/O, retry on connect)
  src/ipc_posix.odin     Unix domain sockets (poll)
  src/scanner.odin       Content scanner: AI-based (LLM), informational alerts only
  src/help.odin          Compile-time help text loading
  src/staleness_test.odin  Tests: staleness TTL, format migration V4→V5
  src/relevance_test.odin  Tests: relevance scoring, counters, feedback, V5→V6 migration
  src/traverse_test.odin   Tests: layered traversal (L0 gates, L1 thoughts, L2 related, budget)
  src/*_test.odin        All tests (odin test src/)
